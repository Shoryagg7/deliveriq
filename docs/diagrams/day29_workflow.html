2 · POST /orders/dispatch — the two-phase claim

every request first passes request_id middleware + the atomic Lua token bucket (shared in Redis)
Phase 1 · scan — lock-free

PostgresLoad all PENDING (plain read)

none → 404 NO_PENDING_ORDERS

AppBuild max-heap

priority = value + wait_minutes × AGING_WEIGHT — aging beats starvation

Why lock-free: FOR UPDATE on the whole set would let one instance lock every order — the others would see an empty queue and falsely 404.
pop best candidate · heap empty → 409
Phase 2 · claim — locks, no mutations

Claim 1 · order rowFOR UPDATE SKIP LOCKED

WHERE id = ? AND status = 'PENDING' — locked-by-other → skipped (never waits); already assigned → filtered by the re-check

✗ row unavailable → ↻ next order from heap
order row is exclusively ours until commit

Redis · pure readselect_rider(exclude=tried)

geohash home+8 cells → haversine → band Δ of nearest eligible → fewest orders_today, then nearest. No writes here (CQS).

band exhausted → ↻ next order

Claim 2 · rider rowFOR UPDATE SKIP LOCKED

WHERE id = ? AND status = 'AVAILABLE' — DB truth re-checked under lock

✗ claim lost → tried.add(rider) → ↻ re-select for the SAME order
The herd fix: concurrent callers all rank the same rider first (differentiating SREM/INCR land post-commit). Abandoning the order on a lost rider measured 5/15 under a 15-burst; the bounded retry loop made it 10/15 — losing a race now costs one candidate, not the whole request.
both rows locked
Phase 3 · mutate — only now

Apptransition · assign · flip

PENDING→ASSIGNED · order.rider_id · rider.status = BUSY

Why last: the session commits the whole unit of work. Mutating before a claim that then fails leaves pending changes a later commit flushes — phantom assignments (the real Day-29 bug).
Phase 4 · commit

Postgresdb.commit()

assignment durable · both row locks released · losers' SKIP LOCKED now sees truth

Phase 5 · Redis catch-up — post-commit only

Redismirror committed reality

SREM rider from geohash cell · INCR orders_today (date-stamped key, 48 h TTL) · publish order.dispatched

Why after commit: a rollback or lost claim must never leave a bumped counter, a hole in the index, or a phantom event. Postgres is truth; Redis mirrors it. Pub/sub stays fire-and-forget — Kafka lands Day 30.
